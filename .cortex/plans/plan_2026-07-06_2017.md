## Datamart: NYC Taxi Star Schema

### Architecture
```
BRONZE (Sources) → SILVER (Staging) → GOLD (Dimensions + Facts)
```

### Nodes to Create

#### SILVER Layer (Stage nodes — truncate & reload)
1. **SILVER-STG_TRIPS.yml** — Union of CAB_TRIPS and YELLOW_CAB_TRIPS with cleaned/cast columns
2. **SILVER-STG_LOCATION.yml** — Pass-through from BRONZE.LOCATION
3. **SILVER-STG_VENDOR.yml** — Pass-through from BRONZE.VENDOR
4. **SILVER-STG_DRIVER.yml** — Pass-through from BRONZE.DRIVERS
5. **SILVER-STG_PAYMENT_TYPE.yml** — Pass-through from BRONZE.PAYMENT_TYPE
6. **SILVER-STG_RATE_CODE.yml** — Pass-through from BRONZE.RATE_CODE
7. **SILVER-STG_CALENDAR.yml** — Pass-through from BRONZE.CALENDAR

#### GOLD Layer (Dimensions — SCD Type 1, with business keys)
8. **GOLD-DIM_LOCATION.yml** — Location dimension (LOCATION_ID as BK, BOROUGH/ZONE/SERVICE_ZONE as change tracking)
9. **GOLD-DIM_VENDOR.yml** — Vendor dimension (VENDOR_ID as BK, VENDOR_NAME as change tracking)
10. **GOLD-DIM_DRIVER.yml** — Driver dimension (DRIVER_ID as BK, name/license/rating as change tracking)
11. **GOLD-DIM_PAYMENT_TYPE.yml** — Payment type dimension (PAYMENT_TYPE_ID as BK)
12. **GOLD-DIM_RATE_CODE.yml** — Rate code dimension (RATE_CODE_ID as BK)
13. **GOLD-DIM_DATE.yml** — Date/calendar dimension (CALENDAR_DATE as BK)

#### GOLD Layer (Fact — MERGE on business key)
14. **GOLD-FCT_TRIP.yml** — Trip fact table (TRIP_ID as BK, FK references to dimensions, measures: fare, tip, tolls, distance, passenger count, total amount, surcharges)

### Implementation Details

- Each node uses V1 YAML format matching the BRONZE source pattern
- Stage nodes use `sqlType: Stage`, Dimensions use `sqlType: Dimension`, Fact uses `sqlType: Fact`
- Dimension columns include `isBusinessKey` and `isChangeTracking` flags
- Fact columns include `isBusinessKey` on TRIP_ID
- Source column references link back to the upstream staging nodes via `sourceColumnReferences`
- The STG_TRIPS staging node will use multisource to union CAB_TRIPS and YELLOW_CAB_TRIPS
